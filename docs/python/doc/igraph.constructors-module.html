generate_edges(formula)

Parses an edge specification from the head of the given formula part and yields the following:

  • startpoint(s) of the edge by vertex names
  • endpoint(s) of the edge by names or an empty list if the vertices are isolated
  • a pair of bools to denote whether we had arrowheads at the start and end vertices

construct_graph_from_formula(Graph, formula= None, attr= "name", simplify= True)

source code 

Generates a graph from a graph formula

A graph formula is a simple string representation of a graph. It is very handy for creating small graphs quickly. The string consists of vertex names separated by edge operators. An edge operator is a sequence of dashes (-) that may or may not start with an arrowhead (< at the beginning of the sequence or > at the end of the sequence). The edge operators can be arbitrarily long, i.e., you may use as many dashes to draw them as you like. This makes a total of four different edge operators:

  • ----- makes an undirected edge
  • <---- makes a directed edge pointing from the vertex on the right hand side of the operator to the vertex on the left hand side
  • ----> is the opposite of <----
  • <---> creates a mutual directed edge pair between the two vertices

If you only use the undirected edge operator (-----), the graph will be undirected. Otherwise it will be directed. Vertex names used in the formula will be assigned to the name vertex attribute of the graph.

Some simple examples:

>>> from igraph import Graph
>>> print Graph.Formula()           # empty graph
IGRAPH UN-- 0 0 --
+ attr: name (v)
>>> g = Graph.Formula("A-B")        # undirected graph
>>> g.vs["name"]
['A', 'B']
>>> print g
IGRAPH UN-- 2 1 --
+ attr: name (v)
+ edges (vertex names):
A--B
>>> g.get_edgelist()
[(0, 1)]
>>> g2 = Graph.Formula("A-----------B")
>>> g2.isomorphic(g)
True
>>> g = Graph.Formula("A  --->  B") # directed graph
>>> g.vs["name"]
['A', 'B']
>>> print g
IGRAPH DN-- 2 1 --
+ attr: name (v)
+ edges (vertex names):
A->B

If you have may disconnected componnets, you can separate them with commas. You can also specify isolated vertices:

>>> g = Graph.Formula("A--B, C--D, E--F, G--H, I, J, K")
>>> print ", ".join(g.vs["name"])
A, B, C, D, E, F, G, H, I, J, K
>>> g.clusters().membership
[0, 0, 1, 1, 2, 2, 3, 3, 4, 5, 6]

The colon (:) operator can be used to specify vertex sets. If an edge operator connects two vertex sets, then every vertex from the first vertex set will be connected to every vertex in the second set:

>>> g = Graph.Formula("A:B:C:D --- E:F:G")
>>> g.isomorphic(Graph.Full_Bipartite(4, 3))
True

Note that you have to quote vertex names if they include spaces or special characters:

>>> g = Graph.Formula('"this is" +- "a silly" -+ "graph here"')
>>> g.vs["name"]
['this is', 'a silly', 'graph here']
Parameters:
  • formula - the formula itself
  • attr - name of the vertex attribute where the vertex names will be stored
  • simplify - whether the simplify the constructed graph
Returns:
the constructed graph:

construct_graph_from_degree_sequence(out, in=None, method="simple", options=None)

source code 

Generates a graph with a given degree sequence, using a given generation method.

Parameters:
  • out - the out-degree sequence for a directed graph. If the in-degree sequence is omitted, the generated graph will be undirected, so this will be the in-degree sequence as well
  • in_ - the in-degree sequence for a directed graph. If omitted, the generated graph will be undirected. Note the name of the argument; in is a reserved word in Python so we cannot use that.
  • method - the generation method to use. Must be one of the following:
    • "simple") -- simple generator based on the configuration model. The enerated graph is not simple and is not guaranteed to be connected. If you want simple graphs only, use "config_model".
    • "fast_random" (legacy alias: "no_multiple") -- similar to "simple" but restarts the generation of the graph if a loop or multiple edge would be created, even in the middle of the generation process. This method is relatively fast and generates simple graphs, but does not sample simple graphs uniformly. Use "config_model" instead if you need to sample simple graphs uniformly.
    • "config_model" -- simple generator based on the configuration model. This option is the same as "simple", but depending on the values of the "loops" and "multiple" options in the options dictionary, it might reject non-simple graphs. When non-simple graphs are rejected (i.e. "loops" and "multiple" are both False or missing), this method samples uniformly from the space of simple graphs.
    • "vl" or "viger_latapy" -- a more sophisticated generator that can sample undirected, connected simple graphs uniformly. It uses Monte-Carlo methods to randomize the graphs. This generator should be favoured if undirected and connected graphs are to be generated and execution time is not a concern. igraph uses the original implementation of Fabien Viger; see the following URL and the paper cited on it for the details of the algorithm: http://www-rp.lip6.fr/~latapy/FV/generation.html.
    • "hh" or "havel_hakimi" -- a deterministic generator that picks two vertices with free edge stubs to connect iteratively, until there are no more stubs left. Variants of the algorithm differ in whether it is allowed to create loop edges or multiple edges, and in the order the vertices are chosen. These can be controlled with a dictionary passed in the options argument. The "loops" key of the options dict controls whether loop edges are allowed (default is False), while the "multiple" key of the options dict controls whether mulitple edges are allowed (default is False). The "order" key of the dict controls in which order the vertices are selected: "smallest" picks the vertices with the smallest number of remaining stubs, "largest" picks the vertices with the largest number of remaining stubs, "index" takes vertices with smaller indices first. The default order is "index". In the undirected case, the "smallest" method is guaranteed to generate a connected graph, provided that a connected realization exists.
  • options - additional options for the generation method; see the documentation of the generation method for more details. Typically, most generation methods accept "loops" and "multiple" as keys; the corresponding values are booleans controlling whether loop and multiple edges are allowed in the generated graph. Other keys are also available for some of the algorithms; refer to the documentation of the method parameter for more details.
